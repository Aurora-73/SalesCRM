## 一、计划总览

### 目标
从 4 个主流开源 CRM 系统中提炼可借鉴的设计思路，分阶段落地到 SalesCRM 中，提升系统的可扩展性、用户体验和 AI 集成深度。

### 阶段划分

| 阶段 | 时间范围 | 核心主题 | 主要产出 |
|------|----------|---------|---------|
| **第一阶段** | 1-2 周 | 小步快跑，快速见效 | 活动时间线 + 多渠道抽象 + 扩展字段 |
| **第二阶段** | 3-4 周 | AI 能力升级 | MCP 接口 + Agent 操作权限 + 工作流规则化 |
| **第三阶段** | 1-2 月 | 架构升级 | 公式引擎 + 技能包系统 + 多视图支持 |
| **第四阶段** | 3-6 月 | 长期演进 | 元数据驱动架构 + RLS 多租户 + 可配置布局 |

## 二、第一阶段：小步快跑（1-2 周）

> **主题**：快速提升用户感知，改动小，见效快

### 1.1 活动时间线（Activity Timeline）

**借鉴来源**：EspoCRM Stream

**需求描述**：
- 为每个联系人生成"关系时间线"，按时间展示互动历史中的关键节点
- 不同于原始消息历史，时间线只提取有意义的事件
- 支持时间线视图和时间跳转

**关键事件类型**：
- 🎉 初次接触（第一次聊天）
- 💬 对话高峰（单日消息量前 10%）
- 🔄 关系升级（信号等级提升）
- 📅 日期节点（认识 N 天、纪念日等）
- 📝 重要笔记（用户手动标记的重要对话）
- 👤 信息更新（档案字段变更）
- ⚠️ 风险预警（关系降温、负面情绪等）

**技术实现**：
- 新增 `compute_timeline(conn, contact_wxid)` 函数
- 基于现有消息和指标数据，生成时间线事件列表
- 在 `brief_data` 或新增 `timeline_data` 结构化工具中返回

**优先级**：🔴 高
**工作量**：小（2-3 天）
**依赖**：无
**相关模块**：`engine/analyzers/metrics.py`、`engine/tools/`

---

### 1.2 多渠道消息抽象（Channel Abstraction）

**借鉴来源**：Twenty CRM Message Channels

**需求描述**：
- 在数据模型层面为未来接入多渠道预留空间
- 当前只有微信，但架构上支持扩展到 WhatsApp、短信、邮件等
- 不改变现有功能，只是为未来铺路

**具体改动**：
- `messages` 表增加 `channel_type` 字段（默认 'wechat'）
- 抽象出 `MessageParser` 基类，当前微信解析器继承它
- 消息分析层不直接依赖微信特定字段，通过通用接口访问

**优先级**：🔴 高（架构层面，越早做越好）
**工作量**：小（1-2 天）
**依赖**：无
**相关模块**：`database/schema.py`、`engine/parsers/`

---

### 1.3 档案扩展字段（Custom Attributes）

**借鉴来源**：Krayin CRM + EspoCRM

**需求描述**：
- 档案字段从"全部硬编码"过渡到"核心字段 + 扩展字段"
- 核心字段（用于分析计算的）固定不变
- 扩展字段由用户自定义，保存在 JSON 字段中
- 不影响现有分析逻辑

**具体设计**：
```python
# 当前：所有字段硬编码
brief_data = {
    "name": "...",
    "age": 25,
    "job": "...",
    # ... 都是固定字段
}

# 改进后：核心字段 + 扩展字段
brief_data = {
    # 核心字段（固定，用于分析）
    "name": "...",
    "gender": "...",
    "age": 25,
    # ...
    
    # 扩展字段（用户自定义）
    "custom_fields": {
        "hobby": "摄影",
        "favorite_food": "火锅",
        "meeting_place": "星巴克"
    }
}
```

**优先级**：🟡 中
**工作量**：小（2 天）
**依赖**：无
**相关模块**：`engine/tools/brief_data.py`、`models/`

## 三、第二阶段：AI 能力升级（3-4 周）

> **主题**：深化 AI 集成，让 Agent 从"分析者"变成"行动者"

### 2.1 MCP 服务器接口

**借鉴来源**：Django CRM (BottleCRM)

**需求描述**：
- 实现 MCP（Model Context Protocol）服务器
- 让 Claude Desktop、Cursor 等 AI 工具可以连接 SalesCRM
- AI 可以直接查询联系人、读取档案、搜索聊天记录

**核心能力**：
- `list_contacts` — 列出联系人
- `get_contact_brief` — 获取联系人档案
- `search_messages` — 搜索聊天记录
- `get_metrics` — 获取关系指标
- `update_brief_field` — 更新档案字段

**安全设计**：
- 使用个人访问令牌（Personal Access Token）认证
- 只读操作默认开放，写操作需要确认
- 遵循最小权限原则

**优先级**：🔴 高（差异化竞争力）
**工作量**：中（3-5 天）
**依赖**：无
**相关模块**：新增 `mcp_server/` 目录

---

### 2.2 Agent 操作权限（Agent as User）

**借鉴来源**：Django CRM + Twenty CRM

**需求描述**：
- 当前 Agent 只能读取数据、给出建议
- 让 Agent 能够执行一些操作，比如：
  - 更新档案字段
  - 添加笔记
  - 设置提醒
  - 标记重要消息
- 操作需要可追溯、可撤销

**安全机制**：
- Agent 操作有明确的审计日志
- 重要操作需要用户确认（两阶段提交）
- 操作范围受限于当前用户权限

**优先级**：🟡 中
**工作量**：中（3-4 天）
**依赖**：无
**相关模块**：`engine/agent/`

---

### 2.3 维护规则配置化（Workflow Rules）

**借鉴来源**：Twenty CRM Workflow + EspoCRM Formula

**需求描述**：
- 当前 `agent/maintain.py` 的维护逻辑是硬编码的
- 将维护规则抽离为可配置的规则列表
- 用户可以自定义维护策略

**规则示例**：
```yaml
maintenance_rules:
  - id: cold_contact_nudge
    name: 冷淡联系人提醒
    trigger: signal_level == "冷淡" AND days_since_last_contact > 7
    action: send_reminder("考虑重新联系一下")
  
  - id: auto_archive
    name: 自动归档
    trigger: days_since_last_contact > 90
    action: mark_archived(true)
  
  - id: birthday_reminder
    name: 生日提醒
    trigger: contact.birthday - today <= 3
    action: send_reminder("生日快到了")
```

**优先级**：🟡 中
**工作量**：中（4-5 天）
**依赖**：无
**相关模块**：`engine/agent/maintain.py`

### 2.4 Agent 执行监控

**借鉴来源**：Twenty CRM AI Agent Monitor

**优先级**：🟢 低
**工作量**：小（2 天）
**依赖**：无
**相关模块**：`engine/agent/`

## 四、第三阶段：架构升级（1-2 月）

> **主题**：提升系统可扩展性，功能模块化

### 3.1 公式引擎升级

**借鉴来源**：EspoCRM Formula Engine

**需求描述**：
- 当前 `formulas.py` 中的公式是硬编码的 Python 函数
- 演进为可配置的公式引擎
- 支持用户自定义指标计算公式
- 类似 Excel 公式的语法

**核心能力**：
- 公式解析器（Parser）
- 内置函数库（数学、统计、文本等）
- 公式执行器（Evaluator）
- 公式编辑器（前端）

**优先级**：🟡 中
**工作量**：大（1-2 周）
**依赖**：无
**相关模块**：`engine/formulas/`

---

### 3.2 技能包系统（Skills System）

**借鉴来源**：Twenty CRM Skills + Krayin Packages

**需求描述**：
- 将功能封装为可插拔的"技能包"
- 每个技能包包含：prompt 模板、工具函数、知识库条目
- 用户可以按需启用/禁用技能
- 支持第三方开发的技能包

**技能包示例**：
- `skill-meeting-analysis` — 会面分析技能
- `skill-gift-recommendation` — 礼物推荐技能
- `skill-conversation-starter` — 话题开场技能
- `skill-conflict-resolution` — 矛盾调解技能

**技能包结构**：
```
skills/
├── date-analysis/
│   ├── skill.json          # 技能元数据
│   ├── prompts/            # Prompt 模板
│   ├── tools/              # 工具函数
│   └── knowledge/          # 相关知识
```

**优先级**：🟡 中
**工作量**：大（1-2 周）
**依赖**：无
**相关模块**：新增 `.claude/skills/` 目录

---

**借鉴来源**：Twenty CRM Views

**优先级**：🟢 低
**工作量**：中（1 周）
**依赖**：无（纯前端）
**相关模块**：前端展示层

## 五、第四阶段：长期演进（3-6 月）

> **主题**：架构级别的改造，支持更大规模的扩展

### 4.1 元数据驱动对象模型

**借鉴来源**：Twenty CRM + EspoCRM

**需求描述**：
- 从"联系人"扩展到更多实体类型时，不重复造轮子
- 所有实体（联系人、公司、商机、任务等）都通过元数据定义
- 统一的 CRUD、搜索、过滤、排序机制

**核心组件**：
- ObjectMetadata — 对象元数据定义
- FieldMetadata — 字段元数据
- RelationMetadata — 关系元数据
- MetadataManager — 元数据管理器

**优先级**：🟢 低（架构升级，不需要马上做）
**工作量**：大（1 月+）
**依赖**：第三阶段完成
**相关模块**：核心架构层

---

### 4.2 行级安全（RLS）多租户

**借鉴来源**：Django CRM (BottleCRM)

**需求描述**：
- 当 SalesCRM 从个人工具升级为团队工具时
- 使用 PostgreSQL RLS 实现数据隔离
- 每个团队/工作区只能看到自己的数据

**优先级**：🟢 低（团队版才需要）
**工作量**：大（2-3 周）
**依赖**：无
**相关模块**：数据库层、认证系统

---

### 4.3 可配置页面布局

**借鉴来源**：Twenty CRM Page Layout

**需求描述**：

- 拖拽式配置，显示哪些模块、什么顺序
- 每个用户可以有自己的布局配置

**优先级**：🟢 低
**工作量**：大（2-3 周）
**依赖**：第三阶段完成

## 六、推荐执行顺序

### 第一优先（马上开始）

## 七、风险与注意事项

### 技术风险
1. **过度设计** — 不要一上来就搞元数据驱动，先满足实际需求
2. **性能问题** — 公式引擎、工作流规则可能影响性能，需要缓存机制
3. **安全隐患** — MCP 接口和 Agent 操作权限需要仔细设计安全边界

### 产品风险
1. **功能膨胀** — 不是所有 CRM 的功能都适合 SalesCRM，聚焦核心价值
2. **用户学习成本** — 技能包、工作流等高级功能可能增加复杂度
3. **维护成本** — 功能越多，维护成本越高，需要平衡

### 建议原则
- **够用就好** — 不追求大而全，解决实际问题最重要
- **渐进式演进** — 从硬编码到可配置，从简单到复杂
- **保持简洁** — 核心功能保持简单直观，高级功能可选开启
