# 方法论 — Wiki 主轴 + 公式辅助 + 冲突裁决 + 指标体系

## Wiki 是推理主轴（贯穿全程）

Wiki 不是"参考资料"，是分析方法论的来源。在分析过程的每个环节都应随时查阅：
- 看到 brief 信号 → 查 Wiki 理解信号含义
- 看到聊天模式 → 查 Wiki 找话术和互动策略
- 看到指标数据 → 查 Wiki 解读指标背后的含义
- 看到关系阶段 → 查 Wiki 找阶段策略
- 看到公式结果 → 查 Wiki 核验而非套阈值
- 写报告时 → 查 Wiki 引用具体条目作为策略依据

使用 `wiki_search("关键词")` 【MCP工具】搜索，`wiki_read("路径")` 【MCP工具】读全文。

## 公式是辅助参考（核验而非套用）

- 公式数值在"数据全貌"表中出现一次
- 后续所有策略判断必须引用 Wiki 条目，不引用公式数值
- BQ=0.5 不一定比 BQ=1.1 差——结合 Wiki 知识核验
- 公式是"视角"不是"裁判"

### BQ 核验示例
`sales_calc_bq` 【MCP工具】返回 BQ=0.5（中性区间）
- ❌ 错误：BQ=0.5 < 0.5 → "敷衍" → "建议放弃"
- ✅ 正确：读 `[[购买意向指标]]` → 客户最近 3 次主动询问产品细节 → 回复率 0.8 → 判断"窗口开放"

## 冲突裁决规则

当不同数据源矛盾时，按以下优先级裁决：

| 优先级 | 来源 | 说明 | MCP 工具 |
|--------|------|------|----------|
| 1（最高） | 实时数据（brief/chat/metrics） | 当前事实，不可推翻 | `person_brief`, `person_chat`, `person_metrics` |
| 2 | 事实档案（evidence: note/date/events） | 客观记录，可信 | `person_evidence`, `person_note`, `person_date_record`, `events_save` |
| 3 | 知识库（wiki_search/wiki_read） | 方法论依据，推理主轴 | `wiki_search`, `wiki_read` |
| 4 | 事件检测（events） | 从数据推导，基本可信 | `events_scan`, `events_save` |
| 5 | 公式计算（formula_*/sales_*） | 量化视角，有启发但需核验 | `formula_calc_*`, `sales_calc_*` |
| 6（最低） | 历史分析（data/outputs/analysis/） | 过去的观点，可能已过时 | — |

## 分析原则

1. **Wiki 先行**：先读 Wiki 找方法论框架，再看数据，最后用公式核验
2. **区分层次**：事实（客户说了什么）→ 推断（客户可能什么意思）→ 建议（你应该怎么做）
3. **引用来源**：分析中引用 Wiki 框架时，标注 `[[条目名]]` 来源
4. **不要泛泛而谈**：每个建议必须具体到"说什么话""什么时候推进""怎么逼单"
5. **注意个人模式**：brief 中的"个人模式警告"要纳入分析
6. **公式只出现一次**：BQ/BSP/IVI/SPE 等数值只在数据全貌表出现，不主导策略分发

## 指标体系速查

16 个加权指标 + signal_flags + media_engagement。通过 `person_metrics(name)` 或 `person_status(name)` 获取。

| 指标 | 含义 | 权重 |
|------|------|------|
| fback | 回复字数比（客户/你） | 0.10 |
| rlatency | 回复速度比（你/客户） | 0.10 |
| fback_quality | 回复质量（正向情绪+追问-敷衍） | 0.10 |
| qscore_personal | 个人化问题比例（意向信号） | 0.10 |
| trend | composite 周变化 | 0.10 |
| escore_volatility | 情绪波动（会话间标准差） | 0.08 |
| moments | 朋友圈互动频率 | 0.06 |
| qscore_functional | 工具化问题比例（供养者信号） | 0.05 |
| rlatency_context | 慢回时有解释的比例 | 0.05 |
| msg_volume_trend | 消息量周变化率 | 0.05 |
| latency_trend | 回复速度周变化率 | 0.05 |
| recent | 最后消息距今天数 | 0.05 |
| active_days | 活跃天数（30 天窗口） | 0.04 |
| escore | 情绪表达比例 | 0.05 |
| msg_count | 消息总数（对数归一化） | 0.02 |

**乘法惩罚**：neediness_penalty（0.4-1.0），消息量比>2 或发起频率>70% 时触发。
**信号等级**：强窗口(≥0.70) / 中窗口(≥0.50) / 弱窗口(≥0.30) / 冷淡(≥0.15) / 无信号(<0.15)
**互动模式**：lover（价值型）/ provider（工具型）/ neutral（中立型）（通过 `person_status` 查看）

## 客户关系阶段（9 个）

| 阶段 | 特征 |
|------|------|
| **未识别** | 联系人存在但尚未分类，无互动数据 |
| **线索** | 有联系方式，尚未正式接触 |
| **初步接触** | 已建立联系，礼貌性沟通，话题浅层 |
| **深入沟通** | 有来有回，主动分享需求，回复较快 |
| **已会面** | 已经线下拜访或会面 |
| **持续跟进** | 会面后继续保持联系，推进方案 |
| **方案推进** | 明确意向信号：主动询价、索要方案、讨论细节 |
| **签约确认** | 已签约或确认成交 |
| **退出/失败** | 基本不再联系，或明确拒绝 |

## Wiki 知识库

覆盖 entity（概念/框架）、topic（主题综述）、synthesis（跨源对比）、scenario（场景决策页）四类页面。

**场景决策页**：客户不回消息了、客户忽冷忽热、被拒绝怎么挽回、价格谈判怎么处理、什么时候该逼单、如何拉升意向、如何走出友谊区（工具型关系转价值型）、第一次会面怎么安排、签约后怎么维护...

**核心框架**：意向指标/意向缺失、关系三要素、70%频率法则、价值型vs工具型、需求感控制、窗口识别、逼单时机、需求挖掘

**Wiki 检索 fallback 策略**（无匹配时）：
| 情况 | Agent 行为 |
|------|-----------|
| 搜索结果为空 | 凭通用商务经验判断，告知用户"Wiki 缺少关于 XX 的条目" |
| 低分命中 | 内容仅供参考，结合事实档案和实时数据独立判断 |
| 关键词未命中 | 用同义词重试（"逼单"→"成交推进"→"报价时机"→"窗口期"） |
