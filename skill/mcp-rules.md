# 规则 — 权限 + 事实档案 + 回复构造 + 路由表 + 禁止事项

## 操作权限规范

| 类型 | 工具 | 是否需要确认 | 说明 | MCP 工具 |
|------|------|------------|------|----------|
| 只读 | brief/chat/metrics/status/rank/wiki/* | ❌ 不需要 | 自由调用 | `person_brief`, `person_chat`, `person_metrics`, `person_status`, `person_rank`, `wiki_search`, `wiki_read` |
| 追加写入 | note/date_record/evaluate | ❌ 不需要 | 直接执行 | `person_note`, `person_date_record`, `person_evaluate` |
| 覆盖写入 | save_from_markdown / person_save_analysis | ⚠️ 覆盖前告知 | save_from_markdown 是必做 | `save_from_markdown`, `person_save_analysis` |
| 检测写入 | events_save | ⚠️ 建议先 scan 展示 | 直接调用会自动检测+写入 | `events_save`, `events_scan` |
| 不可逆 | contact_merge | 🔴 必须确认 | merge 不可撤销 | `contact_merge` |

## 事实档案写入规则

### 概念分层
| 层 | 内容 | 工具 | 冲突优先级 | MCP 工具 |
|---|------|------|-----------|----------|
| 事实档案 | 客观事实 | note / date_record / events | 2（高） | `person_note`, `person_date_record`, `events_save` |
| 分析归档 | 主观判断 | evaluate / save_analysis | 6（最低） | `person_evaluate`, `person_save_analysis` |

### 自检三问（写入 person_note 前必答）
1. 这条信息是客户说的/做的，还是我推断的？→ 只能写前者
2. 如果换一个 Agent 读这条信息，会得出同样的结论吗？→ 如果不会，说明掺杂了判断
3. 这条信息 3 个月后还有效吗？→ 事实是稳定的，判断会过时

### 可以写 vs 不可以写
| ✅ 可以写（客观事实） | ❌ 不可写（主观判断） |
|---------------------|---------------------|
| 客户原话："我们预算 50 万" | "客户肯定要买，值得投入" |
| 客户行为："凌晨主动发消息询价" | "客户一定急着成交" |
| 用户补充："上次会面很成功" | "本月成交概率 80%" |
| 事件记录："6/1-6/15 断联 15 天" | "客户已经不感兴趣了" |

### 特殊说明
`person_evaluate` 虽然写入事实档案文件，但概念上属于分析归档。Agent 读取 `person_evidence` 时，对 evaluations 段落要保持批判性，不能将其与 notes/events/dates 等客观事实同等对待。

## 回复构造规则

### 核心原则
1. 对话领导原则：浅话题（在吗/忙吗/天气）必须 pivot 到需求/价值/深度
2. 每次回复必须有一条可直接发送的消息草稿
3. 话术必须 context-consistent（不编造不存在的叙事）
4. 时间线感知：分析聊天时第一件事梳理时间线
5. 软抗拒 + 正面信号时坚持推进框架

### 时间线梳理要点
- 回复间隔：秒回还是几小时？有没有变化趋势？
- 主动 vs 被动：谁先发的？哪段是客户主动？
- 密集 vs 冷淡：哪段热？哪段降温？降温前发生了什么？
- 最后一条：谁发的？多久没动静了？
- 整体趋势：升温还是降温？

### 话术规则
- 有 hooks：消息要让客户有理由回复（问句/价值点/利益钩子）
- 匹配阶段：线索轻松破冰 / 深入沟通挖需求 / 方案推进谈价值
- 不超过 2 句：简短，不给压力

## 禁止事项（红线）

| 禁止 | 原因 | 正确做法 |
|------|------|----------|
| 跳过 person_sync 直接分析 | 看到的是旧数据 | 先 person_sync 再分析 |
| 只查一次 Wiki | Wiki 是推理主轴，贯穿全程 | 每看到新信息就查 Wiki |
| 分析完不保存 | 历史无法追溯 | 必须调 save_from_markdown |
| 把主观判断写入 note | 事实档案只能写客观事实 | 用 person_evaluate 写判断 |
| 机械套公式阈值 | 公式是参考视角不是裁判 | 结合 Wiki 知识核验 |
| 泛泛而谈的建议 | 不具体就无法执行 | 具体到"说什么话""什么时候推进" |
| 不引用 Wiki 条目的策略 | 策略缺乏依据 | 用 [[条目名]] 格式标注来源 |
